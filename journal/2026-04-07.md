## Notes

### Knowledge-listener scope reduction

arxiv and youtube processors were removed from the listener pipeline entirely:

- **Why:** Both processors already fetched their own content independently (arxiv via the arxiv API + PDF download, youtube via YouTube Data API). The HTML payload written to `~/sync/inbound/` was never used by either processor â€” passing it was pure waste.
- **Decision:** Filter these URL types at the API boundary. `api_server.py` now checks `"arxiv.org" in url.lower() or "youtube.com" in url.lower()` before writing any file. Returns `{"success": true, "skipped": true}` immediately.
- **Listener** now only handles two cases: twitter â†’ `twitter.process_twitter_url()`, everything else â†’ `default.process_url()`.
- `doc_migration.py` was a one-off migration script that imported the now-deleted arxiv processor â€” caught by the code quality reviewer and deleted.
- Removing `marker-pdf` dropped 78 packages from the lockfile (torch, transformers, surya-ocr, etc.). `markdownify` and `anthropic` were promoted to explicit dependencies since they had only been transitive under marker-pdf.

## Web article: OpenAI â†’ Claude Code handoff

Fourth session replaced the OpenAI GPT-4o call in the web article processor with a staged-file handoff to a Claude Code subprocess.

### What changed

**Removed entirely:**
- `call_llm()` and `call_structured_llm()` in `knowledge/utils/llm.py` â€” the GPT-4o structured output path
- All Pydantic models in `webpage.py` (`ExpectedOutput`, `Metadata`, `PublicationInfo`, `Folder`, `DocumentType`, `SourceType`, `ConfidenceLevel`) â€” only existed to parse OpenAI structured output
- `jinja2` template rendering for the article note â€” moved to the skill
- `openai` and `tiktoken` from `pyproject.toml`

**New flow in `process_url()`:**
1. Python does all mechanical work: readability extraction, markdownify, image localization, writes `raw.html` / `readability.md` / `article.md` to `data/web-pages/<url_hash>/`
2. Writes a staged JSON to `data/staged/<url_hash>.json` with `url`, `url_hash`, `readability_markdown`, `web_page_dir` (absolute path), `staged_at`
3. Invokes `claude -p "Process the web article staged at <path> using the web-article skill"` via `subprocess.run(..., check=True)`
4. Claude handles everything downstream: classification, slug generation, note writing, moving the content dir into the vault

**New skill:** `/Users/wkerr/sync/Obsidian/.claude/skills/web-article/SKILL.md`
- 8-step workflow: read staged JSON â†’ classify â†’ generate slug â†’ move content dir â†’ write note â†’ delete staged file
- Both artifacts share the same slug: `<vault>/<folder>/<slug>.md` (note) and `<vault>/<folder>/<slug>/` (content dir with `article.md`, `raw.html`, `readability.md`)
- Same "Highlights for Wes" personalization as the arxiv skill

### Key decisions

- **Slug-based naming** over url_hash: the summary note and content directory are named from the article title (e.g. `why-transformers-work`), not the hash â€” more readable in the vault.
- **Python stays mechanical**: extraction and file I/O stay in Python; all judgment (classification, summarization, tagging) moves to Claude.
- **`call_structured_llm_with_pdf` kept**: the PDF path (Anthropic SDK direct) is unchanged â€” only the OpenAI path was removed.
